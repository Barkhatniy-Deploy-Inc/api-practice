# Наборы данных API (Спецификация)

В этом документе описываются форматы входных параметров (Query/Path — параметры в адресе запроса) и структуры выходных данных (JSON — текстовый формат обмена информацией), которые предоставляет Зебра-стат API. Спецификация базируется на официальном формате открытых данных Госавтоинспекции.

---

## 1. Входные данные (Параметры запросов)

Входные данные передаются преимущественно через Query-параметры (фильтры после знака «?» в URL) в URL.

### 1.1. Базовые параметры (Управление выдачей)
*   `limit` (integer): ограничение количества возвращаемых записей (максимум 1 000).
*   `offset` (integer): смещение для реализации Pagination (постраничного вывода).
*   `format` (string): формат ответа (`json` или `csv` — текстовая таблица).

### 1.2. Временные фильтры
*   `date_start` (строка, формат `DD.MM.YYYY`): начальная дата выборки.
*   `date_end` (строка, формат `DD.MM.YYYY`): конечная дата выборки.
*   `month` (integer, 1–12): фильтрация по конкретному месяцу.
*   `year` (integer): фильтрация по году.

### 1.3. Географические фильтры
*   `reg_code` (integer): код региона (субъекта РФ). Например, 77 (Москва).
*   `district` (string): название района внутри региона.
*   `dor_code` (string): код федеральной трассы.
*   `is_np` (boolean): `true` — только в населенных пунктах (фильтр «Да/Нет»).

## 2. Выходные данные (Форматы ответов)

### 2.1. Справочные данные (Dictionaries — словари)
Возвращаются эндпоинтами группы `/dictionaries/*`. Представляют собой плоские списки.

**Пример ответа:**
```json
{
  "total": 89,
  "data": [
    { "reg_code": 77, "reg_name": "гор. Москва" },
    { "reg_code": 50, "reg_name": "Московская область" }
  ]
}
```

### 2.2. Агрегированные данные (Статистика для Dashboard — аналитических панелей)
Возвращаются эндпоинтами группы `/stats/*`. Предназначены для отрисовки графиков.

**Пример ответа (Динамика по месяцам):**
```json
{
  "report_type": "monthly_trend",
  "dataset": [
    { "month": "Январь", "count": 4200, "pog": 350 },
    { "month": "Февраль", "count": 3800, "pog": 310 }
  ]
}
```

### 2.3. Детализированная карточка ДТП (Raw Data — «сырые» данные)
Содержит полную вложенную информацию о происшествии.

**Структура объекта `info_dtp`:**
```json
{
  "empt_number": "140000008",        // Уникальный номер ДТП
```

## 3. Методология получения и расчета данных

### 3.1. Прямые данные (Direct Data)
Данные транслируются напрямую из базы данных SQLite (легкая файловая БД). Выполняется простая SQL-выборка с фильтрами.

### 3.2. Расчетные показатели (Calculated Analytics — математические расчеты)
Вычисляются на лету (в момент запроса) на стороне сервера.

#### А. Анализ динамики (Тренды)
*   **Формула:** `((V_cur - V_prev) / V_cur) * 100 %` (сравнение текущего значения с прошлым периодом).

#### Б. Коэффициенты тяжести и рисков
*   **Severity Index (Индекс тяжести):** `pog / (pog + ran)` (смертность среди всех пострадавших).
*   **Social Risk (Социальный риск):** `(pog / population) * 100 000` (погибшие на 100 тыс. населения, данные Росстата).
*   **Transport Risk (Транспортный риск):** `(pog / vehicles) * 10 000` (погибшие на 10 тыс. автомобилей).

#### В. Корреляционный анализ (Поиск взаимосвязей)
*   **Метод:** Aggregation (группировка и обобщение данных). Анализ зависимости тяжести ДТП от факторов (погода, освещение).

#### Г. Прогнозирование (Prediction)
*   **Метод:** Linear regression (линейная регрессия — математическое предсказание будущего на основе прошлых данных).

#### Д. Тепловые карты (Hotspots — очаги аварийности)
*   **Алгоритм:** Clusterization (кластеризация — объединение близких по координатам точек в группы).
