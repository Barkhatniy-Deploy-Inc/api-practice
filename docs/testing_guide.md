# Инструкция по\u00A0тестированию API (API Testing Guide)

В\u00A0этом документе приведены правила и\u00A0примеры тестирования функциональности Зебра-стат API.

---

## 1. Стратегия тестирования
Чтобы обеспечить высокое качество образовательного API, мы\u00A0используем трехуровневую модель:
1.  **Unit-тесты:** проверка отдельных функций (математика регрессии, расчеты рисков).
2.  **Интеграционные тесты:** проверка взаимодействия эндпоинтов с\u00A0базой данных SQLite.
3.  **Smoke-тесты:** проверка доступности основных ручек после деплоя.

---

## 2. Инструментарий
*   **pytest**\u00A0— основной фреймворк для запуска тестов.
*   **httpx**\u00A0— асинхронный клиент для запросов к\u00A0FastAPI.
*   **curl** или **httpie**\u00A0— для ручной проверки из\u00A0терминала.

---

## 3. Примеры ручных тестов

### Проверка здоровья системы
```bash
curl -i -X GET "http://127.0.0.1:8000/health"
```

### Доступ к\u00A0реестру ДТП (с\u00A0авторизацией)
```bash
curl -H "X-API-KEY: ваш_токен" \
     -X GET "http://127.0.0.1:8000/api/v1/accidents/?limit=5"
```

### Проверка аналитики
```bash
curl -H "X-API-KEY: ваш_токен" \
     -X GET "http://127.0.0.1:8000/api/v1/analytics/trends/monthly"
```

---

## 4. Ожидаемые результаты
*   **200 OK**: запрос успешен, данные в\u00A0формате JSON.
*   **403 Forbidden**: ключ не\u00A0передан или неверен.
*   **422 Unprocessable Entity**: ошибка валидации (например, передана строка вместо даты).
*   **500 Internal Server Error**: ошибка на\u00A0стороне сервера (сообщите разработчику).

---

## 5. Автоматизированный запуск
Выполните команду в\u00A0корне проекта:
```bash
pytest tests/
```
Все тесты должны завершиться со\u00A0статусом `PASSED`.
