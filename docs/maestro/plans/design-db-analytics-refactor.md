# Дизайн-документ: Рефакторинг БД и расширение Analytics API

## 1. Рефакторинг Базы Данных
- **Сущность `Participant`**: Поля `age` и `experience` становятся `Nullable`, убираем `default=0`. Это предотвратит искажение аналитики (проблема «водителей-младенцев»).
- **Сущность `Vehicle`**: Проверка и фиксация связи `accident_id` с поддержкой `ON DELETE CASCADE`.
- **Сущность `District`**: Добавление уникального поля `code` (например, "CFO") для фильтрации.
- **Миграция**: Поскольку проект находится в стадии разработки, допустима пересборка БД с удалением старой версии.

## 2. Обновление Синхронизации (`sync_gibdd.py`)
- Замена логики `val or 0` на строгую проверку: если значение отсутствует или некорректно, вставляем `NULL`.
- Улучшение обработки ошибок при запросах к API ГИБДД.

## 3. Реализация Analytics API
- **Принцип**: Pure Python (использование стандартной библиотеки `math`). Никаких внешних тяжелых зависимостей вроде NumPy/Pandas.
- **Новые эндпоинты**:
    - `/stats/summary`: Сводные данные.
    - `/stats/regions`: Рейтинги по регионам.
    - `/analytics/seasonality`: Влияние времен года.
    - `/analytics/reports/participant-profile`: Портрет типичного виновника.
- **Оптимизация**: Использование агрегатных функций SQLAlchemy (`func.sum`, `func.count`) для минимизации нагрузки на Python-слой.
