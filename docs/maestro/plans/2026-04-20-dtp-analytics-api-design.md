# Design Document: Зебра-стат API Backend

## 1. Problem Statement
**Цель:** Создать образовательный Backend API для анализа данных о ДТП, служащий источником данных для студенческих фронтенд-проектов.
**Контекст:** Проект для Arch Linux, использующий FastAPI и SQLite. Фокус на простоте обучения и надежности.
**Основные вызовы:** Интеграция с существующей БД коллеги, реализация 40 эндпоинтов, математические прогнозы без сторонних тяжелых библиотек.

## 2. Requirements
### Functional
- Справочники (Регионы, Погода, Типы ДТП).
- Реестр ДТП с фильтрацией (дети, алкоголь, периоды).
- Аналитические расчеты (Тренды, Риски, Прогнозы).
- Защита эндпоинтов через X-API-KEY.

### Non-Functional
- Сообщения об ошибках на русском языке.
- Полная документация в Swagger.
- Минимальное количество внешних зависимостей.

## 3. Approach
**Layered Architecture (Models -> Services -> API):**
- **Models**: SQLAlchemy для связи с БД.
- **Services**: Чистый Python для бизнес-логики и математики.
- **API**: FastAPI роутеры.

## 4. Architecture
- **DB**: SQLite (`DTP_DB.db`).
- **Security**: Middleware для проверки заголовка API-ключа.
- **Data Flow**: Request -> Security Middleware -> Router -> Service -> DB -> Schema Validation -> Response.

## 5. Risk Assessment
- **Схема БД**: Решается гибким слоем моделей SQLAlchemy.
- **Математика**: Решается Unit-тестами для формул.
- **Производительность**: Оптимизация SQL-запросов.

## 6. Success Criteria
- Реализация всех 40 эндпоинтов по ТЗ.
- Успешная интеграция с БД коллеги.
- Swagger на русском языке.
- Простой запуск проекта.
