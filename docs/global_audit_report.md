# Отчет о глобальном аудите системы ZebraStat

**Дата:** 24 апреля 2026 г.
**Объект аудита:** ZebraStat API (DTP Analytics System)
**Статус:** Завершено
**Участники:** Security, Performance, SRE, Documentation Specialists

---

## 1. Резюме (Executive Summary)

В ходе комплексного аудита системы ZebraStat были проанализированы аспекты безопасности, производительности, надежности инфраструктуры и качества документации. 

Система демонстрирует высокую функциональную готовность, однако выявлен ряд критических уязвимостей в механизмах аутентификации и конфигурации контейнеризации. Основные риски связаны с потенциальными атаками на ключи доступа, небезопасным хранением хешей и отсутствием изоляции в Docker-окружении. С точки зрения производительности, текущая архитектура обработки данных в памяти (in-memory clustering) ограничивает масштабируемость и создает риски отказа при росте объемов данных.

Требуется немедленное вмешательство для устранения критических рисков безопасности и пересмотр архитектуры хранения данных.

## 2. Реестр рисков (Risk Register)

### 2.1. Критические риски (Critical)

| ID | Область | Описание | Последствия |
|:---|:---|:---|:---|
| CR-01 | Security | Уязвимость к атакам по времени (timing attacks) в функции `verify_master_key` из-за использования оператора `==`. | Возможность подбора мастер-ключа путем анализа времени ответа сервера. |
| CR-02 | Security | Отсутствие криптографической соли при хешировании. | Уязвимость к атакам по радужным таблицам в случае утечки базы данных. |
| CR-03 | SRE | Запуск Docker-контейнеров под учетной записью root. | Риск полного компрометирования хост-системы при эксплуатации уязвимости в приложении. |
| CR-04 | Performance | Использование кластеризации данных исключительно в оперативной памяти. | Высокий риск потери данных при перезапуске и риск отказа (OOM) при увеличении нагрузки. |

### 2.2. Средние риски (Medium)

| ID | Область | Описание | Последствия |
|:---|:---|:---|:---|
| MR-01 | Performance | Медленная синхронизация данных из-за отсутствия пакетной вставки (batch insert). | Деградация производительности при импорте больших наборов данных. |
| MR-02 | SRE | Отсутствие лимитов ресурсов (CPU/RAM) в конфигурации Docker. | Одна проблемная задача может монополизировать ресурсы сервера, вызвав отказ других сервисов. |
| MR-03 | SRE | Неполный механизм Health-check (отсутствует проверка подключения к БД). | Оркестратор может считать сервис здоровым, даже если он не может выполнять запросы к данным. |
| MR-04 | Security | Избыточные настройки CORS (разрешены лишние домены или методы). | Увеличение поверхности атаки через браузерные сценарии. |

### 2.3. Низкие риски (Low)

| ID | Область | Описание | Последствия |
|:---|:---|:---|:---|
| LR-01 | Performance | Отсутствие индексов в базе данных SQLite. | Замедление аналитических запросов по мере роста объема данных. |
| LR-02 | Docs | Расхождение путей в API: использование `/stats` в коде против `/analytics` в документации. | Путаница для разработчиков-потребителей API. |
| LR-03 | Docs | Отсутствие описания скрипта `seed_db.py` в регламенте Definition of Done (DOD). | Риск потери актуальности процедур инициализации базы данных. |

## 3. План действий (Action Plan)

### Этап 1: Немедленное исправление (Срок: 1-3 дня)
1. **Security**: Заменить оператор `==` на `secrets.compare_digest` в функции `verify_master_key`.
2. **Security**: Внедрить использование соли (salt) при генерации хешей.
3. **SRE**: Обновить Dockerfile для создания не-root пользователя и запуска приложения от его имени.
4. **SRE**: Добавить `healthcheck` в `docker-compose.yml`, включающий проверку доступности порта БД.

### Этап 2: Оптимизация и стабилизация (Срок: 1-2 недели)
1. **Performance**: Реализовать перенос логики кластеризации из памяти в SQL-запросы.
2. **Performance**: Оптимизировать процесс синхронизации, внедрив `executemany` или аналоги для пакетной вставки.
3. **SRE**: Установить `deploy.resources.limits` в Docker Compose.
4. **Performance**: Создать необходимые индексы в SQLite для полей, часто используемых в фильтрации.

### Этап 3: Документирование и комплаенс (Срок: 2-3 недели)
1. **Documentation**: Привести документацию и код к единому стандарту именования эндпоинтов (рекомендуется `/analytics`).
2. **Documentation**: Обновить файл `docs/dod.md`, включив требования по поддержке и документированию скриптов инициализации (`seed_db.py`).
3. **Security**: Пересмотреть политику CORS и ограничить доступ только необходимыми источниками.
