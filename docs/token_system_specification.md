# Спецификация системы токенов и\u00A0безопасности (Token System Design)

В\u00A0этом документе описана архитектура системы доступа, управления API-ключами и\u00A0механизмов авторизации для Зебра-стат API.

---

## 1. Концепция доступа
Доступ к\u00A0API осуществляется на\u00A0основе статических API-ключей (X-API-KEY), передаваемых в\u00A0заголовках HTTP-запросов. Система спроектирована с\u00A0учетом простоты для студентов и\u00A0высокой безопасности для серверной части.

### Технические решения:
*   **Место хранения:** переменные окружения (`.env` файл).
*   **Тип сравнения:** Constant Time Comparison (защита от\u00A0атак по\u00A0времени).
*   **Область действия:** глобальная (защищает все эндпоинты API).

---

## 2. Формат токенов
Токен\u00A0— это строка в\u00A0формате `sk_dtp_practice_<random_hex>`.
*   **Длина:** минимум 32\u00A0символа.
*   **Пример:** `sk_dtp_practice_secure_key_12345678`

---

## 3. Механизм проверки (Security Layer)
Реализуется на\u00A0уровне Middleware или Dependencies в\u00A0FastAPI.

### Логика работы:
1. Перехват заголовка `X-API-KEY`.
2. Если заголовок отсутствует\u00A0— ошибка `403 Forbidden` («Ключ доступа не\u00A0предоставлен»).
3. Если ключ не\u00A0совпадает со\u00A0значением из\u00A0настроек\u00A0— ошибка `403 Forbidden` («Некорректный ключ доступа»).
4. Все попытки доступа с\u00A0неверным ключом логируются (без вывода самого ключа в\u00A0логи).

---

## 4. Группы доступа (Будущее расширение)
Сейчас реализована единая роль Developer. В\u00A0будущем возможна сегментация:
*   `Read-Only`: доступ только к\u00A0GET-запросам (справочники, аналитика).
*   `Admin`: доступ к\u00A0загрузке данных (POST /sync).

---

## 5. Инструкция для студентов
Для выполнения запросов к\u00A0API необходимо добавить заголовок в\u00A0клиент:
```javascript
headers = {
  "X-API-KEY": "ваш_ключ_из_файла_.env"
}
```
