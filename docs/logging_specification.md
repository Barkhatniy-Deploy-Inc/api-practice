# Спецификация системы логирования и мониторинга (Logging & Monitoring)

В данном документе описывается структура сбора логов, форматы сообщений и механизмы отслеживания состояния DTP Analytics API.

---

## 1. Цели логирования
Логирование в системе предназначено для:
*   Отладки работы аналитических модулей.
*   Мониторинга активности пользователей (студентов).
*   Аудита безопасности (отслеживание попыток несанкционированного доступа).
*   Анализа производительности (время обработки запросов).

---

## 2. Архитектура логов
Система использует двухуровневое логирование:
1.  **Консольный вывод (Stdout):** Для мониторинга в реальном времени при разработке.
2.  **Файловое хранилище:** Ротируемые JSON-логи в папке `logs/` для долгосрочного хранения.

### Параметры ротации:
*   **Файл:** `logs/api_access.log`
*   **Макс. размер:** 10 МБ.
*   **Количество бэкапов:** 5 последних файлов.

---

## 3. Формат сообщений (Structured Logging)
Все сообщения записываются в формате JSON для легкого парсинга (например, через `jq`).

```json
{
  "timestamp": "2026-04-22 15:33:51,299",
  "level": "INFO",
  "logger": "api.core",
  "message": "Request: GET /api/v1/accidents/ Status: 200 Latency: 0.0330s"
}
```

### Поля:
*   `timestamp`: Дата и время события.
*   `level`: Уровень (DEBUG, INFO, WARNING, ERROR, CRITICAL).
*   `logger`: Имя модуля (например, `api.analytics`).
*   `message`: Текстовое описание события с техническими деталями.

---

## 4. Категории событий
| Категория | Логика |
| :--- | :--- |
| **HTTP Access** | Метод, путь, статус-код и время ответа. |
| **Security** | Ошибки валидации API-ключа (без вывода ключа). |
| **Database** | SQL-запросы в режиме отладки и ошибки соединения. |
| **Analytics** | Факты успешного построения прогнозов. |

---

## 5. Инструменты анализа
Для работы с логами из консоли Arch Linux рекомендуется использовать:
*   `tail -f logs/api_access.log` — просмотр в реальном времени.
*   `cat logs/api_access.log | jq` — форматированный вывод.
