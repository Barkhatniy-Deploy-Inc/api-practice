# Отчет и руководство по\u00A0тестированию токенов (Token Testing Guide)

В этом документе описаны процедуры верификации системы API-ключей, реализованной в Зебра-стат API.

---

## 1. Методология проверки
Для\u00A0обеспечения безопасности образовательного сервиса мы\u00A0используем автоматизированное тестирование системы авторизации, разделенное на\u00A0три ключевых этапа:
1.  **Проверка жизненного цикла:** генерация -> использование -> отзыв.
2.  **Негативное тестирование:** использование некорректных, коротких или\u00A0чужих токенов.
3.  **Изоляция доступа:** проверка того, что\u00A0обычный ключ студента не\u00A0дает прав на\u00A0генерацию новых ключей.

---

## 2. Сценарии тестирования

### ✅ Сценарий №1: Успешный доступ (Happy Path)
*   **Действие:** отправить запрос `POST /api/v1/auth/keys` с\u00A0валидным `X-MASTER-KEY`.
*   **Ожидаемый результат:** сервер возвращает статус 200 и\u00A0объект с\u00A0полем `access_key`.
*   **Проверка:** использование полученного ключа в\u00A0заголовке `X-API-KEY` для\u00A0любого метода аналитики возвращает статус 200.

### ❌ Сценарий №2: Неавторизованный доступ
*   **Действие:** отправить запрос без\u00A0заголовка `X-API-KEY`.
*   **Ожидаемый результат:** статус **403 Forbidden**.
*   **Сообщение об\u00A0ошибке:** «Ключ доступа не\u00A0предоставлен».

### ❌ Сценарий №3: Попытка взлома Master-функций
*   **Действие:** отправить запрос на\u00A0создание ключа, используя обычный студенческий токен в\u00A0заголовке `X-MASTER-KEY`.
*   **Ожидаемый результат:** статус **403 Forbidden** или\u00A0**401 Unauthorized**.

---

## 3. Автоматизированные тесты (pytest)
Все сценарии реализованы в\u00A0файле `tests/test_auth_system.py`. Запуск производится командой:
```bash
pytest tests/test_auth_system.py
```
**Критерий успеха:** 100% пройденных тестов (Зеленая зона).

---

## 4. Инструментарий для\u00A0ручной проверки
Для\u00A0быстрой проверки токена рекомендуется использовать `curl`:
```bash
# Проверка владельца ключа
curl -I -H "X-API-KEY: ваш_токен" http://127.0.0.1:8000/api/v1/auth/me
```

---

## 5. Рекомендации по\u00A0безопасности
*   **Мастер-ключ** должен храниться исключительно в\u00A0переменных окружения сервера и\u00A0быть длиной не\u00A0менее 32 символов.
*   При\u00A0утере студентом ключа, администратор должен деактивировать старый хеш в\u00A0БД перед\u00A0выдачей нового.
