Полный проект базы данных «DTP Analytics»

Чтобы реализовать все 40 ручек, нам нужно 6 таблиц. Это обеспечит нормализацию и высокую скорость работы.

 * id: INTEGER PRIMARY KEY.

Таблица regions (Справочник регионов)
Нужна для Группы 1 и Группы 2.
 * code: TEXT PRIMARY KEY (Код ГИБДД, например, "77").
 * name: TEXT.
 * district_id: INTEGER (FK -> districts).
Таблица accidents (Центральный реестр ДТП)
Это ядро системы. Сюда стекаются данные для Групп 2, 3, 5 и 6.
 * id: INTEGER PRIMARY KEY.
 * empt_number: TEXT UNIQUE (Системный номер из JSON).
 * Временные метки (Группа 3):
   * date_dtp: DATE (ГГГГ-ММ-ДД).
   * time_dtp: TIME (ЧЧ:ММ).
   * is_holiday: BOOLEAN (Флаг праздничного дня для ручки №19).
 * География (Группа 2):
   * region_code: TEXT (FK -> regions).
   * locality: TEXT (Населенный пункт).
   * is_city: BOOLEAN (1 — в городе, 0 — трасса. Для ручек №10, 11).
   * road_category: TEXT (Федеральная, Региональная. Для ручки №12).
   * is_railway: BOOLEAN (Для ручки №13).
 * Условия и причины (Группа 5):
   * weather: TEXT (Дождь, Туман и т.д.).
   * road_condition: TEXT (Мокрое, Гололед).
   * light: TEXT (Освещение).
   * has_defects: BOOLEAN (Для ручки №34).
   * infrastructure: TEXT (Светофоры, разметка. Для ручки №35).
 * Последствия (Группа 6):
   * fatalities: INTEGER (Поле pog из JSON).
   * injured: INTEGER (Поле ran из JSON).
   * rescue_time: INTEGER (Минуты. Для ручки №40).
Таблица vehicles (Транспортные средства)
Нужна для ручки №28.
 * id: INTEGER PRIMARY KEY.
 * accident_id: INTEGER (FK -> accidents).
 * type: INTEGER FK -> car_type (Легковой, грузовой).
 * brand: TEXT FK -> car_brand.
 * model: TEXT FK -> car_model.
 * year: INTEGER (Год выпуска).
Таблица participants (Участники)
Нужна для Группы 4 и ручки №32.
 * id: INTEGER PRIMARY KEY.
 * accident_id: INTEGER (FK -> accidents).
 * vehicle_id: INTEGER (FK -> vehicles, может быть NULL для пешеходов).
 * role: TEXT (Водитель, Пешеход, Велосипедист).
 * gender: BOOLEAN (1 - мужской пол, 0 - женский)
 * age: INTEGER (Для ручки №26).
 * experience: INTEGER (Стаж v_st. Для ручки №25).
 * is_drunk: BOOLEAN (Для ручки №32).
 * health_status: TEXT (Степень тяжести, можно сделать как выпадающий список).
 * first_aid: BOOLEAN (Оказана ли помощь. Для ручки №39).
2. Реализация 40 ручек: Логика привязки к БД
Здесь описано, как именно БД должна отдавать данные для каждого эндпоинта.
📂 Группа 1: Базовые данные (5 ручек)
 * Список ДТП: Выборка из accidents с LIMIT и OFFSET.
 * Карточка {id}: SELECT * FROM accidents WHERE id = ?.
 * Регионы: SELECT * FROM regions.
 * Показатели: Вывод ENUM-значений (pedestrians, children и т.д.).
 * Погода: SELECT DISTINCT weather FROM accidents.
🗺 Группа 2: Территория (8 ручек)
 * Сводка РФ: SUM(fatalities), SUM(injured) по всей таблице accidents.
 * Округа: GROUP BY district_id.
 * Топ регионов: COUNT(*) с GROUP BY region_code и ORDER BY DESC.
 * Сводка региона: Фильтр WHERE region_code = ?.
 * Города: Фильтр WHERE is_city = 1.
 * Трассы: Фильтр WHERE is_city = 0.
 * Федеральные дороги: Фильтр WHERE road_category = 'Федеральная'.
 * Ж/Д переезды: Фильтр WHERE is_railway = 1.
📅 Группа 3: Время (7 ручек)
 * По годам: GROUP BY year.
 * По месяцам: GROUP BY month.
 * Дни недели: GROUP BY day_of_week.
 * По часам: GROUP BY hour.
 * Будни vs Выходные: GROUP BY (case when day_of_week in (6,7) then 'weekend' else 'workday' end).
 * Праздники: Фильтр WHERE is_holiday = 1.
 * Ночное время: Фильтр WHERE hour BETWEEN 23 AND 05.
👥 Группа 4: Участники и Транспорт (8 ручек)
 * Пешеходы: Поиск в participants где role = 'Пешеход'.
 * Дети: Фильтр WHERE age
< 16.
 * Велосипедисты: Фильтр WHERE role = 'Велосипедист'.
 * Общественный транспорт: Фильтр в vehicles по типу ТС.
 * Стаж: AVG(experience) или группировка по диапазонам.
 * Возраст: Группировка age по категориям (18-25, 25-40 и т.д.).
 * Гендер: GROUP BY gender.
 * Виды ТС: GROUP BY type в таблице vehicles.
🌦 Группа 5: Факторы (7 ручек)
 * Погода: GROUP BY weather.
 * Покрытие: GROUP BY road_condition.
 * Освещение: GROUP BY light.
 * Алкоголь: Подсчет accidents, где в participants есть хотя бы один is_drunk = 1.
 * Скорость: Фильтр по полю нарушений (можно добавить в accidents как cause_speeding).
 * Дефекты дорог: Фильтр WHERE has_defects = 1.
 * Инфраструктура: Группировка по infrastructure.
🏥 Группа 6: Тяжесть (5 ручек)

 * Раненые: SUM(injured).

3. Связи (Relations)
 * districts (1) --- (N) regions
 * regions (1) --- (N) accidents
 * accidents (1) --- (N) vehicles
 * accidents (1) --- (N) participants
 * vehicles (1) --- (N) participants (для водителей и пассажиров).
4. Итоговые рекомендации по созданию
 * Типизация: Для SQLite используй INTEGER для флагов (0 и 1), так как чистого BOOLEAN там нет.
 * Индексы: Обязательно создай индексы на year, region_code и is_drunk. Без них аналитика по 40 ручкам будет «вешать» сервер при большом объеме данных.
 * Автозаполнение: Поля year, month, hour и day_of_week должны заполняться автоматически в Python-коде при парсинге поля date_dtp и time из JSON.
