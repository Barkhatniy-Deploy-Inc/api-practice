Основываясь на предоставленном Техническом задании и структуре JSON-ответа, для перевода вложенных данных ГИБДД в реляционную модель SQLite потребуется процесс нормализации. Поскольку SQLite не имеет встроенной поддержки массивов или сложных JSON-объектов (хотя и имеет JSON-расширения, классический реляционный подход предпочтительнее для аналитики), массивы из JSON (`spog`, `factor`, `ts_info`, `ts_uch`) необходимо вынести в отдельные связанные таблицы.

Ниже представлена полная структура базы данных с выделенными сущностями, полями и связями.

---

### 1. Основная таблица: `accidents` (ДТП)
Это центральная сущность, хранящая базовую информацию о происшествии, его месте и общих последствиях. Объект `location`, `outcome` и скалярные поля из `dor_usl` "сплющиваются" (flatten) в эту таблицу для оптимизации запросов.

* **`id`** *(TEXT, Primary Key)* — Уникальный номер ДТП (соответствует `empt_number`).
* **`date_dtp`** *(TEXT)* — Дата в формате YYYY-MM-DD (для удобной сортировки в SQLite, парсится из `date_dtp`).
* **`time_dtp`** *(TEXT)* — Время ДТП (`time`).
* **`latitude`** *(REAL)* — Широта (`coord_w`).
* **`longitude`** *(REAL)* — Долгота (`coord_l`).
* **`district`** *(TEXT)* — Район (`location.district`).
* **`settlement`** *(TEXT)* — Населенный пункт (`location.np`).
* **`street`** *(TEXT)* — Улица (`location.street`).
* **`road`** *(TEXT)* — Автодорога (`location.dor`).
* **`road_category`** *(TEXT)* — Значение дороги (`location.dor_z`).
* **`km`** *(INTEGER)* — Километр дороги (`location.km`).
* **`m`** *(INTEGER)* — Метр дороги (`location.m`).
* **`accident_type`** *(TEXT)* — Вид ДТП (`dtpv`).
* **`vehicles_count`** *(INTEGER)* — Количество ТС (`k_ts`).
* **`participants_count`** *(INTEGER)* — Количество участников (`k_uch`).
* **`fatalities`** *(INTEGER)* — Погибших (`outcome.pog`).
* **`injured`** *(INTEGER)* — Раненых (`outcome.ran`).
* **`road_surface`** *(TEXT)* — Состояние проезжей части (`dor_usl.s_pch`).
* **`lighting`** *(TEXT)* — Освещенность (`dor_usl.osv`).

---

### 2. Сущность: `vehicles` (Транспортные средства)
Поскольку в одном ДТП может участвовать несколько автомобилей (массив `ts_info`), они выносятся в отдельную таблицу.

* **`id`** *(INTEGER, Primary Key, AutoIncrement)* — Внутренний ID базы данных.
* **`accident_id`** *(TEXT, Foreign Key -> `accidents.id`)* — Привязка к конкретному ДТП.
* **`vehicle_number`** *(INTEGER)* — Порядковый номер ТС в аварии (`n_ts`).
* **`vehicle_type`** *(TEXT)* — Тип автомобиля (`t_ts`, например, "Легковые автомобили").
* **`brand`** *(TEXT)* — Марка (`marka_ts`).
* **`model`** *(TEXT)* — Модель (`m_ts`).
* **`color`** *(TEXT)* — Цвет (`color`).
* **`year_of_manufacture`** *(INTEGER)* — Год выпуска (`g_v`).

---

### 3. Сущность: `participants` (Участники ДТП)
Массив `ts_uch` вложен внутрь ТС. Эта таблица будет хранить водителей, пассажиров и пешеходов. Пешеходы могут быть привязаны напрямую к ДТП, если они не относятся к конкретному автомобилю.

* **`id`** *(INTEGER, Primary Key, AutoIncrement)* — Внутренний ID.
* **`accident_id`** *(TEXT, Foreign Key -> `accidents.id`)* — Прямая привязка к ДТП (для аналитики).
* **`vehicle_id`** *(INTEGER, Foreign Key -> `vehicles.id`, Nullable)* — Привязка к ТС (может быть `NULL` для пешеходов).
* **`role`** *(TEXT)* — Категория участника (`kt_uch`: Водитель, Пешеход, Пассажир).
* **`gender`** *(TEXT)* — Пол (`pol`).
* **`driving_experience`** *(INTEGER)* — Стаж вождения в годах (`v_st`).
* **`health_status`** *(TEXT)* — Состояние здоровья/Тяжесть травм (`s_t`).
* **`alcohol_status`** *(TEXT)* — Состояние опьянения (`alco`).

---

### 4. Словари и связи "Многие-ко-многим" (M:N)
В JSON-структуре присутствуют массивы строк (например, погодные условия `spog: ["Дождь"]` и факторы `factor: ["Сведения отсутствуют"]`). Для них создаются справочники и сводные таблицы.

**Таблицы для Погодных условий (`spog`):**
* **`dict_weather`**: `id` (INTEGER, PK), `name` (TEXT) — Справочник погоды.
* **`accident_weather`**: `accident_id` (TEXT, FK), `
weather_id` (INTEGER, FK) — Таблица связи.

**Таблицы для Дорожных факторов (`factor`):**
* **`dict_factors`**: `id` (INTEGER, PK), `name` (TEXT) — Справочник факторов.
* **`accident_factors`**: `accident_id` (TEXT, FK), `factor_id` (INTEGER, FK) — Таблица связи.

---

### 🔗 Итоговая схема связей (Relationships)

1.  **`accidents` 1 : M `vehicles`**
    * Одно ДТП может включать одно или несколько транспортных средств. Связь через `vehicles.accident_id = accidents.id`.
2.  **`accidents` 1 : M `participants`**
    * В одном ДТП участвует один или несколько человек. Связь через `participants.accident_id = accidents.id`.
3.  **`vehicles` 1 : M `participants`**
    * В одном автомобиле может находиться водитель и несколько пассажиров. Связь через `participants.vehicle_id = vehicles.id`.
4.  **`accidents` M : N `dict_weather`**
    * У одного ДТП может быть несколько погодных условий, одно условие (например, "Дождь") применимо к тысячам ДТП. Реализуется через `accident_weather`.
5.  **`accidents` M : N `dict_factors`**
    * Аналогичная связь для сопутствующих дорожных факторов. Реализуется через `accident_factors`.
