# Техническая спецификация базовой настройки (Core System Design)

В данном документе приведено исчерпывающее описание системной архитектуры DTP Analytics API. Спецификация служит эталоном для реализации модулей конфигурации, безопасности, логирования и документации.

---

## 1. Архитектура конфигурации (Pydantic Settings)

Управление настройками реализуется через библиотеку `pydantic-settings`. Это обеспечивает строгую типизацию, валидацию значений при старте и удобный доступ к `.env` через объект `settings`.

### 1.1. Переменные окружения (.env)
Все параметры разделены на логические блоки. Значения по умолчанию (fallbacks) используются только для некритичных настроек.

| Переменная | Тип | Описание | Пример / Дефолт |
| :--- | :--- | :--- | :--- |
| `PROJECT_NAME` | `str` | Имя проекта в OpenAPI/Swagger | `DTP Analytics API` |
| `VERSION` | `str` | SemVer версия приложения | `1.0.0` |
| `API_V1_STR` | `str` | Базовый префикс всех эндпоинтов | `/api/v1` |
| `X_API_KEY` | `SecretStr` | Статический ключ доступа (защита) | `sk_dtp_...` |
| `AUTH_SALT` | `SecretStr` | Секретная соль для стабильного хеширования API-ключей | `s_...` |
| `DATABASE_URL` | `str` | DSN для подключения к SQLite | `sqlite:///./dtp.db` |
| `LOG_LEVEL` | `enum` | Уровень детализации логов | `INFO` (DEBUG/ERROR) |
| `DEBUG_MODE` | `bool` | Режим отладки (подробные ошибки) | `False` |

### 1.2. Механизм загрузки
1. Модель `Settings` инициализируется при импорте `app.core.config`.
2. Происходит поиск файла `.env` в корне проекта.
3. Валидация: если `X_API_KEY` отсутствует или короче 32 символов, либо `AUTH_SALT` отсутствует или короче 16 символов, приложение должно завершить работу с критической ошибкой.

## 2. Безопасность и Middleware (Security Layer)

### 2.1. Валидация X-API-KEY
Защита реализуется через `FastAPI.Security` (APIKeyHeader) для автоматической интеграции в Swagger.

**Алгоритм обработки запроса:**
1. Перехват заголовка `X-API-KEY`.
2. Если заголовок отсутствует: возврат `403 Forbidden` с телом `{"detail": "Ключ доступа не предоставлен"}`.
3. Сравнение (Constant Time Comparison): использование метода сравнения строк, защищенного от атак по времени.
4. Если ключи не совпадают: возврат `403 Forbidden` с телом `{"detail": "Некорректный ключ доступа"}`.

### 2.2. Исключения (Exclusion List)
Следующие пути являются публичными и не требуют проверки ключа:
*   `/docs`, `/redoc` (документация).
*   `/openapi.json` (схема API).
*   `/health` (проверка состояния сервера для балансировщиков).

## 3. Инженерное логирование (Structured Logging)

Система использует стандартную библиотеку `logging` с надстройкой `RotatingFileHandler`. Цель — получение машиночитаемых логов для анализа инцидентов.

### 3.1. Формат записи (Structured JSON)
Для производственной среды рекомендуется формат JSON, для локальной — текстовый. Каждая запись об HTTP-запросе включает:

```json
{
  "timestamp": "2026-04-15T10:25:01.456Z",
  "level": "INFO",
  "logger": "api.access",
  "request": {
    "method": "GET",
    "path": "/api/v1/stats/summary",
    "ip": "192.168.1.45",
    "user_agent": "Mozilla/5.0..."
  },
  "response": {
    "status_code": 200,
    "latency_ms": 42.5
  }
}
```

### 3.2. Ротация и хранение
*   **Файл**: `logs/api_access.log`.
*   **Условие ротации**: Достижение размера 10 МБ (`maxBytes=10485760`).
*   **Глубина архива**: Хранение 5 последних файлов (`backupCount=5`).
*   **Маскирование**: В логах запрещено хранить значение заголовка `X-API-KEY`.

## 4. Спецификация Swagger UI и OpenAPI

Настройка `FastAPI(docs_url="/docs")` должна соответствовать корпоративным стандартам визуализации.

### 4.1. Группировка и теги
Для навигации используются теги (`tags_metadata`), задающие порядок групп:
1.  **Dictionaries**: Справочники (Regions, Weather, Types).
2.  **Accidents**: Операционные данные (Registry, Detailed cards).
3.  **Analytics**: Расчетные показатели (Trends, Risks, Correlation).
4.  **System**: Метаданные и Health-checks.

### 4.2. Описание ответов
Каждый эндпоинт в Swagger обязан содержать:
*   `summary`: Краткое действие (до 5 слов).
*   `description`: Подробная логика из спецификации (Markdown).
*   `responses`: Описание кодов 200 (успех), 403 (ошибка доступа), 422 (ошибка валидации) и 500 (ошибка сервера).

---

## 5. Глобальная обработка исключений (Error Handling)

Проектируется единый обработчик (Exception Handler) для всех непредвиденных ошибок.
*   **Логика**: Любая ошибка `500` должна записываться в лог с полным StackTrace, но пользователю возвращается только `{"detail": "Внутренняя ошибка сервера. Обратитесь в техподдержку"}`.
*   **Валидация**: Ошибки `Pydantic` (422) перехватываются для перевода сообщений на русский язык (например, "Поле обязательно для заполнения" вместо "Field is required").
